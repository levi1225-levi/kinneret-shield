DATABASE SCHEMA (Prisma)
=======================

Users & Authentication:
  - User                  # Core user table with roles
  - Student               # Student profile + linked parents
  - Parent                # Parent profile
  - Teacher               # Teacher profile + departments
  - Staff                 # Security guards, management

Locations & Devices:
  - Room                  # Physical rooms with capacity
  - Device                # NFC readers (ESP32+PN532)

NFC & Attendance:
  - NfcCard               # NFC card UIDs linked to users
  - AttendanceRecord      # Check-in/check-out records
  - Schedule              # Class schedules
  - ScheduleEnrollment    # Student enrollments

Alerts & Events:
  - Alert                 # System alerts with severity
  - EmergencyEvent        # Lockdown/evacuation events
  - DailyReport           # Attendance summaries

Operational:
  - InviteCode            # Registration invites (7-day expiry)
  - DeviceLog             # Device operation logs
  - Announcement          # System announcements
  - AuditLog              # User action audit trail
  - Notification          # User notifications
  - Settings              # System configuration

AUTHENTICATION FLOW
===================

1. Registration:
   - User provides invite code during signup
   - Invite code validated & marked as used
   - Role-specific profile created (student/teacher/staff)
   - User created with Google ID

2. Login:
   - Google OAuth 2.0 callback
   - User validated/created
   - JWT token generated (24h expiry)

3. API Requests:
   - Bearer token in Authorization header
   - JWT strategy validates token
   - @Roles() decorator checks permissions
   - RolesGuard enforces access control

4. Device Authentication:
   - Base64-encoded (deviceId:secret)
   - DEVICE_AUTH_SECRET verified
   - Device record validated

NFC TAP WORKFLOW
================

1. Device reads NFC card UID
2. POST /attendance/tap/:deviceId with cardUid
3. System:
   - Validates NFC card exists & active
   - Resolves student from card
   - Checks current check-in status
   - If checked in: checkout
   - If checked out: check in (with capacity check)
   - Broadcast via WebSocket
   - Generate alerts if needed

4. Response includes:
   - success/failure status
   - Student name & ID
   - Action (check_in/check_out)
   - Timestamp

SCHEDULED TASKS
===============

Every hour:
  - Auto-checkout students (1-hour timeout)

Every 5 minutes:
  - Check device heartbeats
  - Mark offline if missed (2-minute timeout)

Daily at midnight:
  - Generate daily reports for all rooms
  - Calculate: total check-ins, unique students, avg duration

ALERT SYSTEM
============

Auto-triggered:
  - unknown_card: NFC card not registered
  - capacity_exceeded: Room at max capacity
  - device_offline: Device missed heartbeat
  - emergency: Emergency event initiated

Manual:
  - Any type created by security/management
  - Can be resolved with user audit trail
